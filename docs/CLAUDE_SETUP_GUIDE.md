## 1. Rules (Automatic Guidance)

**Location**: `.claude/rules/`  
**Trigger**: Automatically load based on file patterns  
**Purpose**: Guide Claude's behavior and enforce standards

| Rule | When Loaded | Purpose |
|------|-------------|---------|
| `modernization.md` | Session start | .NET 10 + C# 13 standards |
| `testing.md` | Editing test files | 80% coverage enforcement |
| `spec-first.md` | Editing src/**/*.cs | No code without specs |
| `github-workflow.md` | Session start | Issue/wiki process |

### Key Enforcement

- **spec-first.md** stops code changes without:
  1. Failing test (written first)
  2. GitHub issue with acceptance criteria
  3. ECMA standard reference
  4. Pure refactoring (no behavior change)

## 4. Workflows (Multi-Agent Orchestration)

**Location**: `.claude/workflows/`  
**Trigger**: Explicit user request or ultracode mode  
**Purpose**: Coordinate multiple agents for complex tasks

### When to Use

✅ Use workflows when:
- Task requires 10+ independent operations
- Need parallel agent execution
- Adversarial verification needed
- User explicitly requests orchestration

❌ Don't use workflows for:
- Simple linear tasks (use skills)
- Exploratory work (scout first)
- Tasks one agent can handle

### Example Workflow

```javascript
// Modernization workflow (generated by Claude)
export const meta = {
  name: 'fix-nullable-violations',
  description: 'Fix nullable reference type errors across all files',
  phases: [
    { title: 'Scan' },
    { title: 'Fix' },
    { title: 'Verify' }
  ]
};

// Phase 1: Scan for issues
phase('Scan');
const issues = await agent('Find all nullable violations', {schema: ISSUES_SCHEMA});

// Phase 2: Fix in parallel
phase('Fix');
const fixes = await parallel(issues.files.map(file => () =>
  agent(`Fix nullable warnings in ${file}`, {isolation: 'worktree'})
));

// Phase 3: Verify
phase('Verify');
const buildResult = await agent('Build and verify no warnings');
return { success: buildResult.succeeded, fixedFiles: fixes.length };
```

## 6. MCP Servers (External Tools)

**Location**: `.mcp.json`  
**Purpose**: Integrate external APIs and databases

### Available (Disabled by Default)

| Server | Purpose | Requires |
|--------|---------|----------|
| `github` | Create issues, PRs | `GITHUB_TOKEN` env var |
| `filesystem` | Advanced file ops | None (built-in sufficient) |
| `sqlite` | SQLite database | Database file |

### Why Disabled?

- Adds overhead to every session
- Built-in tools cover most needs
- Enable only when specifically needed

### How to Enable

Edit `.mcp.json`:
```json
{
  "mcpServers": {
    "github": {
      "disabled": false  // Change to false
    }
  }
}
```

Set environment variable in `.claude/settings.json`:
```json
{
  "environment": {
    "GITHUB_TOKEN": "your_token_here"
  }
}
```

## 7. Worktree Include

**Location**: `.worktreeinclude`  
**Purpose**: Copy gitignored files to worktrees

### What Are Worktrees?

Git worktrees are isolated copies of the repo. Used when:
- Multiple agents edit files simultaneously (prevents conflicts)
- Testing changes without affecting main directory

### What to Include

Files that are gitignored but needed in worktrees:
- `.env` files (if you create them)
- Local configuration
- Test data

## Permissions Configuration

**File**: `.claude/settings.json`

### Current Permissions

```json
{
  "permissions": {
    "allow": [
      "Bash:**",       // All bash commands (safe ones)
      "Read:**",       // Read any file
      "Write:**",      // Write any file
      "Edit:**",       // Edit any file
      "Glob:**",       // File pattern matching
      "Grep:**"        // Content search
    ],
    "deny": [
      "Bash:rm -rf /*",     // Prevent destructive commands
      "Bash:curl *",        // No external downloads
      "Bash:ssh *",         // No remote access
      // ... other dangerous commands
    ]
  }
}
```

### Hooks

```json
{
  "hooks": {
    "afterEdit": [
      {
        "description": "Verify code builds after C# edits",
        "command": "dotnet build src/ECMABasic.sln --nologo --verbosity quiet",
        "when": { "glob": "**/*.cs" }
      }
    ]
  }
}
```

After every C# file edit, the build runs automatically to catch errors immediately.

## Usage Patterns

### Daily Development

1. **Start work**: Rules automatically guide you
2. **Check coverage**: `/verify-coverage`
3. **Review code**: Delegate to `code-reviewer` agent
4. **Write tests**: Delegate to `test-writer` agent
5. **Create issues**: `/create-modernization-issue "Title"`
6. **Build audit**: `/audit-build` to see progress

### Large Refactoring

1. **Plan**: Create GitHub issues with acceptance criteria
2. **Scout**: Explore the codebase manually
3. **Orchestrate**: Request workflow if 20+ files
4. **Execute**: Workflow coordinates multiple agents
5. **Verify**: Each agent validates its work
6. **Aggregate**: Results combined into report

### Code Review

```
You: "Review the changes in CharacterReader.cs"
Claude: [Spawns code-reviewer agent]
Agent: [Checks standards, architecture, tests]
Claude: [Reports findings with specific feedback]
```
